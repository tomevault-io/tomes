# How Claude should treat this vault

> [!important] **NEVER add `# Title` headings to any note.** Obsidian shows the filename as the title; a duplicate H1 is redundant. Start every file with frontmatter, then go straight into body content (callouts, prose, lists). This applies to ALL notes — no exceptions.

This vault is the user's "second brain": meetings, people, projects, summaries, daily notes. AI agents working here read from and write to it constantly. Re-read this file before adding new conventions and update it if you introduce new folders/patterns during a session.

## Folder structure

| Folder | Purpose |
|---|---|
| `01 Updates/` | Daily research updates by topic (one file per topic). Newest entries at top, older days collapsed into `> [!note]-` callouts grouped by `## YYYY-MM`. |
| `02 Daily/YYYY/MM/` | Daily notes, named `MM-DD-YY ddd.md` (e.g. `03-29-26 Sun.md`). |
| `03 Meetings/` | Call/meeting notes, organized by year (`2025/`, `2026/`...), named `MM-DD-YY Day Name1 x Name2.md`. Transcripts under `03 Meetings/Transcriptions/`. |
| `04 People/` | Person notes. Simple people: flat `Full Name.md`. People with attached docs (style guides, transcripts): folder `Full Name/` with `Full Name.md` as the root note. |
| `05 Projects/` | One folder per project. Each has a root note `<projectName>.md` with frontmatter `project`, `summary`, `started`, `updated`, `links`, `dependencies`, `status`. |
| `06 Research/` | Deep research documents. Frontmatter: `created`, `updated`, `topic`, `type: research`. |
| `07 References/` | Clippings, articles, external knowledge, tool docs. |
| `08 Summaries/` | Summary notes generated by the `summarize` and `summarize-call` skills. |
| `10 Places/` | Place notes (cafes, restaurants, gyms, hotels, homes). |
| `Archive/` | Completed or inactive projects — move folders here instead of deleting. |
| `_Attachments/` | All media files (audio, images, PDFs, ebooks). |
| `_Bases/` | Obsidian Bases (`.base` files). Optional — only if you use the Bases plugin. |
| `_Templates/` | Reusable note templates. |

When a project is created or updated, also update `PROJECTS.md` at the vault root to keep the dashboard current.

## Daily note format

- Path: `02 Daily/YYYY/MM/MM-DD-YY ddd.md`
- No `# Title` heading (filename is the title)
- Organize work under `##` headers by project/topic (e.g. `## vault setup`, `## summarize skill bugfix`)
- Bullets under each — terse, link to relevant notes
- Always include `## calls/meetings` if any occurred
- `## TODO for next session` at the bottom for carryover

**Mandatory**: at the end of every session that touched the vault, update or create the daily note. Work that isn't logged here is invisible.

## Person notes

- One note per person, in `04 People/<Full Name>.md`
- Top-level callouts:
  - `> [!note] current age: \`=date(today)-date(this.birthday)\`` — Dataview inline query (skip if Dataview isn't installed). If birthday is estimated, append `(estimated)` to the callout text — but `birthday` in frontmatter must remain a pure YAML date.
  - `> [!info]` — rich snapshot of who they are: life story, mission, current focus, key context. Update as new info emerges.
- `## updates` section — chronological log of interactions, each linking to the call/meeting/episode note where they appeared. **Never link to daily notes from `## updates`** — link the source content note.
- Person notes grow organically. For briefly mentioned people, only add what's known. Don't fill out empty boilerplate sections.

### When to create person notes

For every meeting, call, or piece of summarized content:

- **Every named person mentioned gets a note.** Don't leave wikilinks pointing to nonexistent notes.
- **One layer deep**: for every person physically present, also create notes for anyone *they* mention by name (family, friends, colleagues). Not recursive — one hop only.
- **Private individuals** (friends, family): minimal note with only what's known from the conversation.
- **Public figures** (creators, athletes, executives): research and write a substantive `> [!info]` snapshot — birthday, career, links, key facts. Not a stub.
- **Family/partner extraction is mandatory**: when someone's spouse/partner/children/parents/siblings are mentioned, populate the `partner` and `family` frontmatter fields AND create person notes for each (even with placeholders like `[[<Name>'s Son]]` if names aren't given).

### Aliases and disambiguation

- Use display aliases for relationships in frontmatter: `[[John Doe|older brother]]`, `[[Jane Smith|partner]]`.
- Never create two separate wikilinks for the same entity. Use alias syntax: `[[Cobie|Jordan Fish]]`, not `[[Cobie]] / [[Jordan Fish]]`. One entity = one link target.
- Same for renamed companies/products: `[[Facebook|Meta]]`, `[[X|Twitter]]`.

## Meeting/call note conventions

- Hyperlink ALL people, places, and notable nouns (companies, products, concepts) — even if the note doesn't exist yet.
- Body has `## Summary` with key points. **Never repeat frontmatter in the body** — no `# Title`, no recording link if `recording:` is in frontmatter, no transcript link if `transcript:` is in frontmatter.
- Add a `## People mentioned` section listing everyone referenced with a one-line context.
- Transcript notes (in `03 Meetings/Transcriptions/`): frontmatter only for metadata; body is ONLY the timestamped transcript lines.

## Templates (in `_Templates/`)

- **Person**: see [[new person template]]. The `summarize` and `summarize-call` skills install this on first run.
- Add your own templates for books, places, projects, calls, video notes, etc. as your workflow demands. Keep them minimal — frontmatter + a couple of section headers.
- All templates use `{{date:YYYY-MM-DDTHH:mm}}` for `created`/`updated`.

## Bases (optional — Obsidian Bases plugin)

If you use the Bases plugin, store `.base` files in `_Bases/`. Common ones:

- `meetings.base` — all calls/meetings, with per-person filtered views
- `posts.base` — all videos/podcasts/articles (the `summarize` skill writes here)
- `books.base` — all books

When a new person appears in summarized content (podcast episode, YouTube video, etc.), add a per-person view to `posts.base` and embed it in their person note:

```yaml
- type: table
  name: "Person Name"
  filters:
    and:
      - recording != null
      - people.contains(link("Person Name"))
  order:
    - date
    - views
    - file.name
    - summary
  sort:
    - property: date
      direction: DESC
```

Then in their person note: `![[posts.base#Person Name]]`.

For people with real 1-on-1 calls (not just podcast appearances), do the same with `meetings.base` and embed `![[meetings.base#Person Name]]` in a `## meetings/hangouts` section.

## Running logs

For ongoing activity within a project (edits, fixes, deploys), use a single running log note rather than one note per event:

```
- [[MM-DD-YY ddd]] - concise description of what was done
```

Group entries by month with `## YYYY-MM` headers as the log grows. The daily note link provides full context if needed.

## Rules

- **No `# Title` headings** — never. Filename is the title.
- **Never repeat frontmatter in the body.** Frontmatter = metadata; body = content that adds new information.
- **Prefer backlinks over tags** for relationships. Use tags sparingly.
- **Never delete or rename files** without explicit approval.
- **Unread tracking**: when you create or modify any `.md` file in this vault, set `unread: true` in its frontmatter. The bundled `unread-dot` Obsidian plugin shows a blue dot in the file explorer for unread notes and clears it when the user opens them. If the file has no frontmatter, add some.
- **Daily note update is mandatory** at the end of every session — see "Daily note format" above.
