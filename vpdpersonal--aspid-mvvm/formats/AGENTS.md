# Aspid.MVVM

A high-performance Source Generator-based MVVM framework for Unity. Enables clean separation between View, ViewModel, and business logic with zero reflection in bindings and minimal memory allocations.

## Key Technologies

- **C# / .NET Standard 2.0** ‚Äî core framework
- **Roslyn Incremental Source Generators** ‚Äî code generation at build time
- **Unity 2022.3+** ‚Äî target runtime
- **Zenject / VContainer** ‚Äî DI framework integrations
- **UniTask** ‚Äî async utilities
- **.NET 9.0 SDK** ‚Äî for generator/analyzer projects (`global.json`)

## Architecture

The framework uses Source Generators to eliminate boilerplate and reflection:

1. **`[ViewModel]`** ‚Äî marks a `partial` class for source generation; generates `IViewModel` implementation
2. **`[Bind]`** ‚Äî marks fields for binding generation; supports `BindMode` argument
3. **`[RelayCommand]`** ‚Äî transforms a method into an `IRelayCommand` property
4. **Binders** ‚Äî MonoBehaviour components that connect View UI elements to ViewModel bindable members
5. **Observable Collections** ‚Äî thread-safe, change-notifying collections with filtering/sorting

## Code Conventions

**Naming:**
- `PascalCase` for classes, interfaces, methods, properties
- `camelCase` for local variables and parameters
- `_camelCase` prefix for private fields
- `I` prefix for interfaces (`IViewModel`, `IRelayCommand`)
- `Binder` suffix for binding components, `Attribute` suffix for attributes

**Namespaces:**
- Root: `Aspid.MVVM`
- Sub-namespaces: `Aspid.MVVM.StarterKit`, `Aspid.Collections.Observable`
- Editor code lives in `.Editor` sub-namespaces

**Code Style:**
- All classes requiring source generation must be `partial`
- Explicit access modifiers on all members
- Nullable reference types (`T?` for optional values)
- XML documentation comments on all public APIs
- ReSharper suppression comments on generated/special code

## Commands

Generator/analyzer solutions build independently of Unity:

```bash
# Build source generator (produces DLL consumed by Unity)
dotnet build Aspid.MVVM.Generators/Aspid.MVVM.Generators.sln

# Run generator tests
dotnet test Aspid.MVVM.Generators/Aspid.MVVM.Generators/Aspid.MVVM.Generators.Tests/

# Run analyzer tests
dotnet test Aspid.MVVM.Analyzers/Aspid.MVVM.Analyzers/Aspid.MVVM.Analyzers.Tests/
```

The Unity project is opened via the Unity Editor (2022.3+); there is no CLI build script.

## Gotchas

- **Generator artifacts are committed.** Built DLLs (`Aspid.MVVM.Generators.dll`, `Aspid.MVVM.Analyzers.dll`, `Aspid.MVVM.Unity.Generators.dll`) live under `Aspid.MVVM/Packages/tech.aspid.mvvm/` and are checked into the repo. After changing a generator, rebuild its solution and commit the refreshed DLL ‚Äî Unity consumes the DLL, not source.
- **Submodules are required.** Cloning without `--recurse-submodules` leaves the generator/analyzer repos empty; Unity won't compile. `Aspid.Collections` is consumed as a UPM git package (not a submodule).
- **`.NET 9.0 SDK` is pinned** via `global.json` in each generator/analyzer root. Older SDKs fail to build.
- **Classes using `[ViewModel]`, `[Bind]`, `[RelayCommand]` must be `partial`** ‚Äî the generator emits a second partial; without the modifier you get compile errors instead of generated members.

## Git Submodules

The project uses 3 git submodules. After cloning, initialize with:

```bash
git submodule update --init --recursive
```

Submodules:
- `Aspid.MVVM.Generators`
- `Aspid.MVVM.Analyzers`
- `Aspid.MVVM.Unity.Generators`

`Aspid.Collections` is consumed as a UPM git package (`tech.aspid.collections`).
