# AGENTS.md — 仓库执行约束（种子模板）

> 本文件为 AI 辅助开发定义核心约束与防踩坑指南，确保开发行为可预测、可审计、可重复。  
> 使用时请将 `<...>` 占位符替换为项目实际值，并根据项目语言和框架调整具体命令。

---

## 1. 核心原则

- **沟通语言**：全程使用中文沟通。答案直接收束当次诉求，不附加不必要的后续建议（如“下一步可以……”）。
- **事实优先**：所有结论必须基于代码事实、测试结果、版本控制记录，不得凭印象或推测。
- **Debug‑First（禁止静默降级）**：
  - 禁止为“先跑通”而添加隐藏回退、静默容错、默认降级路径。
  - 禁止通过 Mock 或模拟构造虚假成功路径（例如模板化的 `(mock) ok`、吞没异常后继续）。
  - 优先暴露真实失败（显式错误 / 异常 / 日志 / 失败测试），以便定位根本原因。
  - 若确需提供回退机制（仅限安全、隐私或用户明确要求），必须同时满足：显式开启、文档可追溯、可关闭、事先与用户达成一致。
- **阶段目标**：先实现核心逻辑的行为等价（即输入输出一致），再考虑性能优化与基准测试。

## 2. 任务工作流

### 2.1 唯一任务来源与锁定

- 项目使用 `<任务跟踪文件>`（例如 `issues.csv`、`tasks.md` 等）作为**唯一任务驱动源**。临时计划文档（如 `PLAN.md`）仅用于阶段性归档，不作为开发驱动。
- 当临时文档与代码事实冲突时，优先级顺序为：`<任务跟踪文件>` 状态 > `git log` 提交证据 > 当前代码实现。
- 进入任务循环前，必须先确认 `<任务跟踪文件>` 与临时文档认知一致，然后锁定该任务。
- **每次只处理一个原子任务**，循环流程：**读取 → 锁定（状态置为“进行中”）→ 开发 → 验证 → 自审 / 回退 → 标记完成 → 提交 → 取下一任务**。
- 筛选规则：优先选取状态为“未开始”且标签为优先级的任务；若为空，按文件顺序选取第一个“未开始”的任务，并在提交说明中记录处理依据。
- **禁止并行开发多个任务**。

### 2.2 任务文件结构规范（以 CSV 为例）

- **禁止使用百分比进度**，状态必须使用文本枚举。
- 建议表头：`ID, 标题, 内容, 验收标准, 审查要求, 状态, 标签`。
- 状态仅三种：`未开始` / `进行中` / `已完成`。
- 标签可自定义，例如：`待确认` / `高优先级` / `待审查`。
- 每一行必须包含可执行边界：
  - `内容`：明确修改位置 + 修改理由。
  - `验收标准`：可验证、可复现、客观可判定。
  - `审查要求`：明确需要检查的动作（如“验证日志格式”、“确认无死锁”）。

### 2.3 自审与回退（每个原子任务结束前必做）

- 对照 `验收标准` 逐条确认。
- 对照 `审查要求` 逐项检查无遗漏。
- 运行最小相关的测试，并记录测试结果。
- 使用工具（如 `git diff`）确认无任务范围外的残余改动。
- 仅当所有检查通过后，才允许进入下一个任务。

### 2.4 Code Review 模式

- 当任务标题包含 `[Code Review]` 时，角色切换为严格审计模式（技术负责人视角）。
- 审计流程：
  ① 基于 `git diff` 或目标文件读取变更；
  ② 严格对照项目预定义的审查清单（如 `docs/review_checklist.md`）与 `AGENTS.md`；
  ③ 执行可用的类型检查与测试。
- 产出：一份审计报告（例如 `docs/reviews/CR-{ID}.md`）。若发现严重问题，需在任务跟踪文件中追加修复任务（状态置为“未开始”）。
- 审计任务默认**不修改业务代码**，仅输出报告与后续修复条目。

## 3. 质量红线

### 3.1 开发边界

- 禁止为迎合测试而硬编码假设。验收标准必须基于第一性原理推导。
- 基于第一性原理分析影响范围，只修改当前原子任务所需的文件。
- 不顺手修正无关问题（可记录，但不在当前任务内处理）。
- 修改后必须执行当前任务定义的最小充分验证，不以“看起来没问题”代替测试。

### 3.2 工程质量基线

- 遵循 SOLID、DRY、关注点分离、YAGNI 原则。
- 命名清晰、抽象务实；仅在关键或非直观处添加简洁注释。
- 非兼容性要求场景下，避免保留过时兼容分支与死代码。
- 显式处理边界条件，不得隐藏失败。
- 业务逻辑优先依赖注入，不在核心流程中硬编码具体实现。
- 优先使用不可变数据结构；不修改函数入参与隐式全局状态，返回新值表达状态变化。

### 3.3 代码复杂度硬约束

| 指标         | 上限          | 超限处理           |
|--------------|---------------|--------------------|
| 函数长度     | ≤ 50 行（不含空行） | 拆分辅助函数       |
| 文件长度     | ≤ 300 行           | 按职责拆分         |
| 嵌套深度     | ≤ 3 层             | 使用卫语句 / 提前返回 |
| 位置参数     | ≤ 3 个             | 改为配置对象       |
| 圈复杂度     | ≤ 10               | 拆分分支逻辑       |
| magic number | 0                  | 抽取命名常量       |

### 3.4 代码输出纯文本约束（强制）

- **严格禁止**在代码、注释、日志字符串中使用 Emoji、Unicode 装饰符号（如实心三角、圆点、星号变体）及任何非标准 ASCII 装饰性字符。
- 代码注释仅使用纯文本（中文/英文） + 标准注释符号（`//`、`#`、`/* */`）。
- **原因**：部分编辑器/终端显示异常、影响 diff 审查、破坏跨平台兼容性。
- **示例**：
  ```
  错误: // [勾号] 用户认证成功 [靶心]
  正确: // 用户认证成功

  错误: # [叉号] 参数验证失败
  正确: # 参数验证失败
  ```
- Markdown 文档（`.md`）同样适用：用 `-` / `*` / 数字列表代替装饰符号，用 `**加粗**` 代替图标强调。

### 3.5 安全基线

- 严禁在源码中硬编码密钥/凭证，必须使用环境变量或密钥管理机制。
- 数据库访问必须使用参数化查询，禁止字符串拼接 SQL。
- 所有外部输入（用户、API、文件）必须在边界处校验与净化。
- 会话中用户粘贴密钥用于调试属于正常流程；**仅当密钥被写入源码文件**时才应触发“泄漏风险”告警。

## 4. 测试体系

### 4.1 测试代码放置位置（以 Rust/Python 为例）

- **禁止**在核心源码目录（如 `src/`）内新增或保留 `#[test]`、`#[cfg(test)]` 等测试代码。
- 测试代码只能放在：
  - Rust 集成测试目录：`tests/`（可包含子目录如 `tests/unit_tests/`）
  - Python 测试目录：`test/` 或 `tests/`
- 为 `src` 中私有逻辑补充测试，必须通过公开 API 在 `tests/` 侧构造场景。
- **提交前扫描命令**（以 Rust 为例）：
  ```bash
  rg -n "#\[test\]|#\[cfg\(test\)\]" src/ -g "*.rs"
  # 期望：无命中
  ```

### 4.2 验证基线

- 代码应可测试，尽可能使用可重复的自动化验证。
- 后端单元测试执行必须设置合理超时（如 60 秒），防止卡死阻塞 CI。
- 优先使用静态检查、格式化和可复现验证，而非主观判断。
- 主线回归验证记录应沉淀到 `docs/reviews/` 目录，包含执行的命令、退出码、通过/失败摘要及跳过原因。

### 4.3 关键数据格式的口径验证（踩坑通用化）

- 当处理结构化数据（如 CSV、JSON、Protobuf）时，必须明确定义每一列/字段的**顺序、类型、业务含义**，并编写格式校验测试。
- 若数据源存在多版本格式，必须显式区分并注明处理逻辑。
- 常见踩坑：列数不匹配、字段错位、单位不一致、空值处理不当。应在测试中覆盖这些情况。

### 4.4 Schema‑Sensitive 数据库测试规则（强制）

- 只要业务逻辑**依赖真实数据库的列类型、SQL 类型转换、驱动序列化行为或跨数据库主从链路**，就不得仅用 Fake / Stub / 纯内存对象宣称“已覆盖”。
- 此类逻辑必须至少补充 **离线 Schema 契约测试**：
  - 锁定关键 SQL 片段（例如 `varchar → timestamp` 的显式转换）。
  - 区分不同分支（如最新数据分支 vs 历史时间戳分支）。
  - 防止“语义测试通过，但真实 Schema / 类型契约失败”的情况。
- 若当前环境无法连接真实数据库，则必须同时满足：
  - 离线测试明确标注“已覆盖语义，未覆盖真实 Schema”。
  - 在任务跟踪文件或门禁文档中显式标记需要真实环境复验。
  - **不得将 `cargo test` 全绿误认为“真实环境已通过”**。
- 典型踩坑：某字段在真实数据库中是 `varchar` 而代码中当作 `timestamp` 处理，导致运行时类型错误。此类问题必须通过 Schema 契约测试 + 真实门禁双重兜底。

### 4.5 最新认知索引（约定）

当需要理解“当前阶段代码真相”时，优先查阅：

- 全阶段总览：`docs/reviews/CR‑STAGE‑ALIGNMENT‑<日期>.md`
- 当前全局状态：`docs/reviews/CR‑ISSUES‑CURRENT‑STATE‑<日期>.md`
- 各阶段审计报告：`docs/reviews/CR‑PHASE*‑<日期>.md`

规则：

- `AGENTS.md` 仅保留强约束与长期规则。
- 阶段性详细认知统一沉淀到 `docs/reviews/`。
- 若两者冲突，以 `<任务跟踪文件>` 和代码事实为准，并回写文档。

## 5. 领域约束（泛化模板）

### 5.1 核心量纲统一（强制）

- 全链路必须使用**业务定义的标准单位**（如“手数”、“股数”、“张”），不得混用其他单位（如名义金额）。
- 转换系数（例如 `volume_multiplier`）**仅用于对外展示或输入解析**，不得参与内部核心逻辑（如持仓计算、订单量纲判断）。
- 核心 API 返回值和参数必须与标准单位一致。
- **踩坑场景**：内部持仓缓存与业务逻辑单位不一致，导致风控条件永远无法满足，但底层撮合仍在运行。
- 排查路径：追踪数据从写入到读取的完整链路，确保单位转换仅发生在边界。

### 5.2 事件流语义与可观测性（强制）

- 主数据链路的投递语义（如 `at‑least‑once` / `at‑most‑once`）必须明确定义，不得私自改变。
- 幂等键应使用全局唯一标识（如 `会话ID + 序号`）构造，确保重试不会产生重复。
- 失败路径必须**可观测、可告警、可重试**。至少暴露指标：失败总数、重试次数、重复数、缓冲深度等。
- 消费端去重逻辑应统一依据幂等键。
- 当前阶段不引入重型机制（如 WAL、ack checkpoint），优先保证链路可视化诊断且主链行为不变。

## 6. 环境与编译（通用模板）

### 6.1 网络与服务依赖

- 若项目依赖内部数据库或服务，需明确其地址与端口，并确保开发环境可访问。
- 示例：数据库 `<db_host>:<db_port>`、消息队列 `<mq_host>:<mq_port>` 等。
- 测试环境可能需要 VPN 或专线连接，应提前说明。

### 6.2 编译与运行环境（以 Rust/Python 混合项目为例）

| 项目         | 要求                                     | 验证命令示例                     |
|--------------|------------------------------------------|----------------------------------|
| Rust 工具链  | rustc 1.75+ / cargo                       | `rustc --version && cargo --version` |
| Python       | 3.8+（abi3 兼容）                         | `python3 --version`              |
| 虚拟环境     | `venv_<平台>`（如 `venv_linux`、`venv_win`） | `source venv_linux/bin/activate` |
| 构建工具     | maturin                                   | `maturin --version`              |
| 服务连通性   | 依赖服务端口可连接                         | `nc -zv <host> <port>`           |

**编译检查**：
```bash
cargo check
```

**单元测试**（以 Rust 为例）：
```bash
cargo test --test <单元测试模块>
```

**本地开发安装**（Rust/Python 混合）：
```bash
source venv_<平台>/bin/activate   # 激活虚拟环境
unset CONDA_PREFIX                 # 避免与 Conda 冲突
maturin develop --release
```

**打包 wheel**：
```bash
maturin build --release
# 可选：添加时间戳副本
cp target/wheels/<生成的wheel文件> target/wheels/<自定义备份文件名>
```

**生效确认**：检查当前 Python 实际加载的扩展模块路径
```python
import <包名>, <包名>._core as core
print("package =", <包名>.__file__)
print("_core =", core.__file__)
```

**常见避坑**：
- Windows 平台使用 Git Bash，路径格式用 Unix 风格，避免 `set PATH=...` 的 cmd 语法。
- 若 `VIRTUAL_ENV` 和 `CONDA_PREFIX` 同时存在，maturin 可能拒绝执行，需 `unset CONDA_PREFIX`。
- 若出现 `Failed to set rpath ... Failed to execute 'patchelf'`，可尝试 `pip install patchelf` 后重试。
- 编译优化（如 `lto = true`）可能延长首次编译时间，增量编译会变快。

### 6.3 编译后生效提醒

- 对 Rust/Python 混合项目，修改 Rust 代码后**必须重新执行 `maturin develop --release`**，否则 Python 仍会加载旧版扩展，导致“代码已改但测试未生效”的假象。
- 本地回测应优先使用虚拟环境中的开发安装，而不是仅打包 wheel。

## 7. 提交与文件卫生

- 每个原子任务单独提交，提交信息清晰说明改动内容。
- 提交前运行 `git diff --name-only` 确认仅包含任务边界内的文件。
- 默认不提交临时文件、日志目录、编辑器缓存、个人配置文件（除非任务显式要求）。
- 建议在 `.gitignore` 中明确忽略以下模式：`*.log`、`*.tmp`、`__pycache__/`、`.DS_Store`、`venv_*/`、`target/` 等。
