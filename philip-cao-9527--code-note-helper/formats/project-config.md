# 当前项目 AGENTS.md

## 高优先级的Windows 与编码命名要求
- 我是中文用户，并且主要在 Windows 系统下开发。
- 修改 JS / HTML / CSS / JSON / Markdown / manifest / README 等文件时，必须特别注意编码问题。
- 不要引入乱码、BOM 混乱、中文注释异常、换行风格混乱等问题。
- 修改前尽量保持原文件编码与原有风格；新建文本文件默认使用 UTF-8。
- 所有改动后的文件都要保证在 Windows 环境下可以正常编辑、加载和运行。
- 新增文件、目录、测试和报告命名要体现职责与场景，避免只有时间戳、缩写或模糊命名。命名应让后续维护者从文件名看出用途、影响范围或问题场景；例如修复报告可以在版本和日期后追加简短主题说明，而不是只生成一个无法区分内容的名称。

## 超长代码生成约束（必须遵守）
- 曾出现过在修改`service-worker.js`时一次性生成超过 1000 行代码，导致输出上限与可维护性问题。
- 从现在开始，如果代码逻辑非常复杂、预计会生成很长代码，必须**优先分拆为多个文件**，不要一次性堆到单文件。
- 分拆策略不要写死，需按工程原则动态判断，至少评估以下维度：
  1. 职责边界是否清晰（单一职责、低耦合、高内聚）。
  2. 变更扩散范围是否可控（避免一个文件承载多条无关链路）。
  3. 测试与回归是否可隔离（单元测试与手测能按模块验证）。
  4. 复用与演进成本是否更低（后续新增功能不会继续堆积在同一大文件）。
  5. 回滚颗粒度是否足够小（单模块问题可独立回退）。
- 可选分拆角度包括但不限于：按职责层、按业务域、按站点、按运行阶段（初始化/调度/渲染/通信）等；以最小风险和最高可维护性为准。
- 禁止一次性生成超长大段代码（尤其是单文件数百到上千行新增），避免代码冗余、评审困难和回滚困难。
- 本节为最高优先级实现约束：当与其他实现偏好冲突时，优先遵守本节。

## 必读开发文档
- **在开始任何修改之前，必须先阅读 `DEVLOG.md`**（开发日志与架构参考）。
- `DEVLOG.md` 包含：项目架构说明、目录结构、关键约束（站点隔离/共享边界/注入顺序/全局命名空间）、数据存储结构、版本历史与历史决策记录。
- 这份文档是理解项目全貌的最快路径，可以避免重复踩坑和破坏已有架构。

## 语言与输出
- 全程使用简体中文：包括说明、计划、总结、测试记录、README 修改、代码注释。
- 允许保留英文的场景仅限：代码、命令、报错原文、文件路径、域名、API 字段名、专有名词。
- 不要输出英文风格的小标题，如 Thoughts、Plan、Next steps；统一使用中文表达。
- 生成的 plan、task、walkthrough、测试说明、修复说明等内容也必须使用中文。

## 项目修改总原则
- 按“最小化修改 / 不破坏旧功能”的原则实现本次变更。
- 优先复用现有模块、现有调用链、现有数据结构，不要为了局部问题做大范围重构。
- “最小化修改”不是盲目保守：如果是面向真实用户需求、解决反复多次仍未修好的 bug，或为了贴合真实工业场景与业务实践，允许进行范围更大的重构或修改；前提是先说明必要性、影响范围、回归方案，并保持职责边界、测试闭环和可回滚性清晰。
- 修改前先识别入口文件、注入顺序、站点隔离边界、公共模块依赖，再动手改代码。
- 改动后不要只停留在“代码看起来合理”，而要主动完成“修改 → 测试 → 发现问题 → 继续修复 → 再测试”的闭环。

## Chrome Web Store 审核友好约束（涉及扩展审核相关改动时必须遵守）
- 只要本次任务涉及 `manifest.json`、权限、`host_permissions`、`content_scripts`、后台网络请求、远程接口、README / 隐私政策 / 商店说明、用户可见权限说明，就必须先从“是否会增加审核时间和审核风险”的角度自检，再修改。
- 默认目标不是“功能先堆上去”，而是“在满足功能的前提下，尽量减少审核成本、权限范围和解释成本”。
- 不要无故新增权限，不要无故扩大匹配范围，不要无故增加远程访问站点；新增前必须先回答清楚：
  1. 为什么必须要这个权限 / 站点
  2. 是否已有权限可复用
  3. 是否可以改为更精确的匹配模式
  4. 是否可以改为由用户显式触发后再执行

### 对 `host_permissions` 和 `content_scripts.matches`
- 严禁无依据改成 `<all_urls>`、`*://*/*`、`https://*/*` 这类广泛权限。
- 能精确到域名就不要放大到泛域名；能精确到路径就不要只写到整站；能只覆盖真正支持的页面，就不要把无关页面也加进去。
- 新增站点时必须同步检查：该站点是否真的有用户可见功能、README / 隐私政策 / 商店说明里是否能清楚解释、测试是否覆盖。

### 对 `permissions`
- 只保留真实用到的权限；如果某项能力可以通过现有权限完成，不要额外申请更强权限。
- 如果某项能力可以改成用户手势触发后的临时访问（例如更接近 `activeTab` / 可选权限思路），就不要默认申请更广的持久访问。
- 新增会触发权限警告的权限前，必须评估它是否会让现有用户在更新后看到新的授权提示，以及是否值得这项审核与安装成本。

### 对远程访问与远程代码
- 允许请求用户自己配置的 LLM API、WebDAV 等远程接口，但这些远程返回内容必须作为“数据”处理，不得作为扩展代码执行。
- 严禁引入远程托管代码（Remote Hosted Code），包括但不限于：远程 JS / WASM、动态插入远程 `<script>`、对远程返回字符串使用 `eval`、`new Function` 或其他等价执行方式。
- 新增第三方依赖或外部资源时，要先排查编译产物里是否混入远程代码痕迹，避免因为 Blue Argon / MV3 相关问题拖长审核或被拒。

### 对单一用途与功能边界
- 所有新增功能都必须服务于当前扩展清晰、可解释的主用途，不要把互不相关的能力硬塞进同一个扩展。
- 如果新增能力会显著改变扩展用途、数据访问范围或支持站点范围，必须在总结里明确提示这是“可能增加审核成本”的改动。

### 对 README、隐私政策、商店说明与权限理由
- 只要本次改动影响了权限、支持站点、数据流向、远程请求、用户数据处理方式，就必须同步检查 README、`PRIVACY POLICY.md`、发布页权限说明、隐私字段说明是否一致。
- 说明必须具体、真实、可审核，不要写模糊空话；尤其要明确“访问哪些站点、为了什么用户可见功能、数据默认存在哪里、何时会离开本地、是否经过开发者服务器”。
- 不要宣称“不会访问 / 不会收集”与实际实现不一致的内容；不要把“用户主动触发”和“默认自动执行”混写。

### 对 Google 开发者中心商店文案的关键字限制
- 仅当生成或修改 Google 开发者中心里的对外审核文本时适用，包括但不限于：商品详情、简短说明、隐私权页面、权限理由、数据使用说明、审核补充说明。
- 生成代码、`manifest.json`、README、`PRIVACY POLICY.md`、CHANGELOG、DEVLOG、docs 报告、测试文件或项目内部文档时，不适用本小节的“少列站点/品牌名”限制；这些文件仍应按真实代码和文档一致性准确列出必要站点、权限和证据。
- 商店文案中避免密集、重复、列表式堆叠站点名或品牌名，尤其不要写成类似 `LeetCode China/International Sites`、`CodeFun2000`、`Deep-ML`、`TorchCode`、`ChatGPT`、`Gemini`、`Claude` 这类连续品牌清单。此前曾因这类密集品牌关键词触发 Chrome Web Store 的 `Keyword Spam` / `Yellow Argon` 违规。
- 商店文案应优先使用功能类别表达，例如“受支持的在线编程练习页面”“受支持的深度学习代码练习网页”“受支持的 AI 对话页面”，只在审核表单明确要求或用户明确要求时，才少量、必要、非重复地出现具体站点名。
- 权限理由里如果必须解释 host 权限，不要堆完整站点清单；应说明访问范围服务于哪些用户可见功能、何时读取页面内容、是否仅限支持页面、数据是否离开本地。
- 如果用户要求输出商品详情或隐私权表单文案，必须先自检是否存在关键字堆砌风险；若存在，先改为自然、概括、非重复的表达。

### 审核友好自检（涉及上述内容时必须执行）
1. 是否新增了任何权限、host、content script 匹配范围或远程访问域名？
2. 新增内容是否都能在代码、README、隐私政策和商店说明中相互对应？
3. 是否存在可以收窄的匹配模式、可以删除的旧权限、可以延后到用户触发时再申请 / 执行的能力？
4. 是否存在让审核员难以快速理解的“大而全”逻辑、无用代码、未完成入口、死代码或难以解释的兼容分支？
5. 是否误引入了远程代码、动态执行、不可审计逻辑或与当前单一用途无关的功能？


## 禁止无依据的防御性编程 / 埋雷式保护逻辑
- 禁止为了“看起来更稳”而加入没有明确需求依据的硬编码限制、截断规则、时间阈值、重试上限、静默回退或隐藏兜底。
- 典型禁止项包括但不限于：
  - 无依据的固定超时（例如 30s / 60s / 90s 后直接报错或中断）
  - 无依据的文本长度截断（例如题目描述、Prompt、参考答案、消息内容超过某个字数就截断）
  - 无依据的条数上限、分页上限、节点数量上限
  - 无依据的重试次数限制、轮询次数限制
  - 无依据的“默认成功 / 默认空字符串 / 默认跳过”式兜底
  - 为了避免报错而偷偷吞异常、提前 return、悄悄降级
- 只要是新增限制，就必须先满足以下条件之一，否则不要加：
  1. 产品需求或 README 已明确要求
  2. 真实接口 / 平台 / 浏览器环境有客观限制
  3. 代码中已有同类约束，并且本次只是沿用现有约定
  4. 已通过测试或证据证明：不加这层限制会稳定导致故障、卡死、崩溃或严重体验问题

## 新增长度 / 超时 / 上限时的强制要求
- 如果确实必须新增以下任何一类限制：
  - 超时
  - 长度上限
  - 条数上限
  - 重试上限
  - 截断逻辑
  - 降级逻辑
  - 回退逻辑
  那么必须同时满足以下要求：
  1. 在代码注释中明确写出“为什么必须有这层限制”
  2. 在最终总结中单独说明该限制的触发条件、目的、影响范围
  3. 在 fix report 中记录这条限制的背景、测试方式和风险
  4. 优先做成可配置项、可观测项，或者至少保证后续容易调整
  5. 不得改变正常成功路径，不得让合法内容被静默截断，不得让长输出在无明确依据时被中途判失败

## 对 API、流式输出、题目抓取的特别约束
- 对 API 流式输出：
  - 不要凭空增加固定超时后直接失败的逻辑。
  - 除非接口协议、平台限制或用户明确要求，否则不要因为“等待较久”就擅自中断生成。
  - 需要处理的是“取消、断网、服务端报错、状态码异常、真正无响应”，不是“时间稍长”本身。
- 对题目描述、题解、参考答案、Prompt 抓取：
  - 不要无依据地按字数、段落数、节点数截断。
  - 应优先保证抓取完整性，再处理展示层优化。
  - 展示层若必须折叠，应使用“UI 折叠/展开”，而不是“数据层直接裁掉”。
- 对 DOM 抓取与页面渲染：
  - 不要因为某个节点暂时没出现，就立刻判定失败并永久放弃。
  - 优先根据真实页面生命周期处理延迟渲染、局部重绘、异步挂载，而不是写死脆弱阈值。

## 错误处理原则
- 错误处理应以“显式暴露问题、便于排查”为目标，不要为了避免报错而伪造成功状态。
- 禁止：
  - broad try/catch 后什么都不做
  - catch 后直接返回空结果冒充成功
  - 失败后自动切成另一套逻辑却不提示
  - 将真实错误伪装成“正常无数据”
- 推荐：
  - 保留必要日志
  - 暴露真实错误原因
  - 区分“用户取消 / 网络失败 / 状态码错误 / 服务端异常 / 页面结构变化”
  - 在不破坏正常链路的前提下做最小必要兜底
